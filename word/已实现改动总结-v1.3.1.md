## 6. 原文件预览功能重构（前端直连对象存储）

彻底解决原文件预览问题，方案核心：**前端直连对象存储公开 URL，按文件类型分发渲染，服务器零开销**。

### 改动文件

| 文件 | 改动 |
|---|---|
| `frontend/package.json` | 新增 `docx-preview` 依赖 |
| `frontend/src/utils/previewUtils.ts` | 新增：可预览类型白名单 + `getPreviewStrategy` / `isPreviewable` 工具函数 |
| `frontend/src/components/FilePreviewModal.tsx` | 新增：预览模态框组件，按 contentType 分发渲染策略 |
| `frontend/src/pages/ResumeDetailPage.tsx` | 重构：预览按钮条件显示（不可预览类型隐藏）、点击打开 Modal、移除旧 `ms-word:` 协议逻辑 |

### 渲染策略

| 文件类型 | 渲染方式 | 服务器开销 |
|---|---|---|
| PDF / 图片 / TXT | iframe 直接加载 storageUrl | 零 |
| DOCX | fetch blob → docx-preview 库渲染 | 零 |
| Markdown | fetch 文本 → react-markdown 渲染 | 零 |
| HTML | fetch 文本 → sandbox iframe srcdoc | 零 |
| DOC（旧格式）/ 其他 | 隐藏预览按钮，仅保留下载 | — |

### 设计要点

- 利用对象存储 bucket 已设置为 public 的事实，文件流量完全绕过后端。
- 后端零改动：`ResumeDetailDTO` 已有 `storageUrl` + `contentType` 字段。
- HTML 预览使用 `<iframe sandbox="allow-same-origin">` 隔离脚本执行。
- 不可预览类型（DOC 旧格式等）自动隐藏预览按钮，避免用户点击后无响应。

## 7. 技术债清理（6.1 / 6.2 / 6.3）

### 7.1 远程节点 ChatClient 缓存清理（6.1）

- 文件：`AiRuntimeCacheInvalidationNotifier.java`
- 改动：`evictLocal(snapshotKey)` 方法在清理 snapshot 缓存后，补充调用 `llmProviderRegistry.evictAllChatClients()`
- 原因：远程节点只收到 snapshotKey 字符串，无法反推完整的 ChatClient 缓存 key（需要 providerId + model + fallback + baseUrl + configVersion），因此采用全量清理策略
- 影响：ChatClient 创建成本低（仅包装 ChatModel），over-evict 对性能无实质影响

### 7.2 大文件流式返回（6.2）

| 文件 | 改动 |
|---|---|
| `FileStorageService.java` | 新增 `downloadFileAsStream(fileKey)` 方法，使用 `s3Client.getObject()` 返回 `ResponseInputStream<GetObjectResponse>` |
| `ResumeHistoryService.java` | `getOriginalFile` → `getOriginalFileAsStream`，返回 `OriginalFileStreamResult(InputStream, contentLength, filename, contentType)` |
| `ResumeController.java` | 返回类型从 `ResponseEntity<byte[]>` 改为 `ResponseEntity<InputStreamResource>`，Content-Length 从 S3 响应元数据获取 |
| `ResumeControllerTest.java` | 测试适配新的流式 API，使用 `ByteArrayInputStream` 模拟 |
