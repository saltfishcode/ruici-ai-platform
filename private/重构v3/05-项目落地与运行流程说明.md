# 项目落地与运行流程说明（10 行执行版）

1. 先执行 `docker/postgres/init.sql`；老库补齐用 `private/重构v3/04-AI运行时配置初始化与补偿脚本.sql`。
2. 任何控制面结构问题先修 SQL，再谈 Java 映射；不把关键结构寄托给 `ddl-auto=update`。
3. 执行顺序必须固定：`SQL -> Entity/Repository -> Resolver/Policy/Snapshot -> Registry缓存 -> Chat -> Embedding -> Voice`。
4. 解析口径统一：`策略校验 -> 请求覆盖(仅允许接口) -> 快照/缓存 -> DB -> 静态配置 -> 代码默认`。
5. 禁止业务模块自行拼接优先级；统一走 `AiRuntimeConfigResolver`。
6. 禁止高热链路每次查库；配置变更后只做受影响 key 的局部失效。
7. Embedding 仅允许任务/批次级快照，禁止 chunk 级动态切换。
8. Voice 仅允许会话级 LLM 快照，禁止进行中会话中途漂移 provider/model；`ASR/TTS` 仍保持静态稳定配置。
9. 测试门禁：先过 resolver/policy/registry 单测，再过 DB 命中 + 覆盖 + 故障降级集成测试。
10. 每阶段完成后更新 `AI模型动态配置执行计划.md` 与 `06-验收清单与测试模板.md`，未更新视为未完成。

## 禁止项（执行时直接判失败）

- 在数据库中存储任何 API Key、Token、密码。
- 放开任意业务接口直接透传 provider/model。
- 使用无冲突目标的 `ON CONFLICT DO NOTHING`。
- 为了“快”跳过测试直接推进下一阶段。
